# 《你的 Markdown 渲染器，扛得住 AI 输出吗？》逐页演讲词

预计时长：18–20 分钟。正文可以直接照着讲；`【操作】` 和 `【过渡】` 不需要念出来。

现场只需要记住整场的一句话：**AI 流式输出不是把静态 Markdown 多渲染几次，而是一种新的更新模型。**

## 第 1 页｜开场：你的 Markdown 渲染器，扛得住 AI 输出吗？

大家好，我是 Simon。

今天我想先问大家一个问题：你的 Markdown 渲染器，真的扛得住 AI 输出吗？

过去我们把 Markdown 当成一种已经写完的文档格式：拿到完整字符串，解析一次，再展示出来。但 AI 输出不是这样。它高频地吐出 chunk，内容越来越长，中间还可能出现代码、图表、公式，甚至工具调用和业务组件。

所以今天我想分享的，不只是一个 Markdown 组件，而是一个我在做 AI UI 时越来越明确的判断：当模型返回得更快、内容更长、形态更复杂，Markdown 渲染器已经从一个展示组件，变成了产品基础设施。

【过渡】在展开这个问题之前，我先用半分钟介绍一下自己。

## 第 2 页｜关于我

我是一名前端开发者，也是一名开源维护者，目前在上海。过去几年，我在 Vue 社区和开源社区比较活跃。自己维护的项目包括 markstream-vue、vscode-use、awesome-compressor 等，也参与过 UnoCSS、DeepChat、Vue Vine 等开源项目。

在 AI 还没有这么普及的年代，也就是大家现在调侃的“古法编程”时代，我还做过很多 VS Code 插件。那时候没有 Agent 帮我们写代码，我就通过插件解决编码过程中的各种小问题，希望把开发体验做得更顺手一点。

后来，我来到现在的公司做 AI Agent 开发，开始真正面对模型持续输出、复杂内容渲染和页面交互稳定性这些问题。也正是在这个过程中，markstream-vue 逐渐产生了。

所以回头看，我之前做的这些事情其实有一条很清晰的主线：以前是通过编辑器插件改善“人怎么写代码”，现在则是在解决“AI 生成的内容，怎样稳定地呈现给人”。

【过渡】为什么偏偏是现在，我们需要重新看待 Markdown 渲染？先看模型这一侧发生了什么。

## 第 3 页｜为什么是现在

这几年，模型的上下文窗口从 8K、128K，一路走到 256K，甚至 1M 以上，量级放大了大约 128 倍。

上下文变长本身并不会直接压垮浏览器，真正的问题是几件事叠加在一起：输出上限更大、chunk 更密、每秒 token 更多，而且输出里出现了更多复杂 UI block。

最终，这些工作都会落到浏览器主线程上。对于 60 帧的页面，每一帧只有 16.7 毫秒；Markdown 解析、Vue 更新、DOM 布局，以及用户的滚动和输入，都在争抢这同一份预算。

所以模型能力越强，前端面对的不是单纯“多一点文本”，而是一种完全不同的持续更新压力。

【过渡】这会带来一个有点反直觉的结果：模型越快，浏览器反而越可能先卡住。

## 第 4 页｜浏览器成为瓶颈

一段 AI 回复从模型出来以后，首先要经过 Markdown 解析，得到段落、列表、代码块等结构化数据。接下来，前端可能还要加工这些数据，把不同内容映射成文本、代码块或业务组件；然后由 Vue 完成组件渲染，更新 DOM 并触发页面布局，最终才变成用户看到的界面。

这条链路里，常见的问题有三个。

第一，CPU 追不上，因为每来一个增量，都可能重新解析整篇长文档。第二，DOM 规模不断增长，布局和绘制越来越重。第三，也是用户感知最明显的：更新会打断滚动、文本选择和输入，页面开始抖动，甚至丢状态。

因此我们的目标不能只是“把内容渲染出来”，而是要在内容还在生成时持续更新，同时保住滚动、选择和组件状态。

【过渡】要理解问题怎么产生的，我们先回到传统 Markdown 渲染器的工作方式。

## 第 5 页｜传统渲染方式

传统的渲染方式非常直接：把完整的 Markdown 文本交给 markdown-it，生成 HTML，然后写入页面容器。

对于博客、文档和 README，这种方式非常合适，因为输入本来就是已经写完的内容。解析一次，展示一次，简单而且稳定。

但 AI stream 给我们的不是完成稿，而是一连串不断增长、不断变化的中间态。如果每收到一个 chunk，都把当前全文重新交给 parser，再完整替换 DOM，我们实际上是在用“渲染完成稿”的方式处理“持续编辑中的文档”。

这不是 markdown-it 做错了，而是内容进入页面的方式已经变了。

【过渡】下一页我们直接看这个过程。

## 第 6 页｜Demo：markdown-it 全量渲染

【操作】点击“播放全量”，停顿两秒，让观众先看左侧源码和右侧结果一起增长。

现在左边是持续进入的 Markdown 源码，右边是 markdown-it 的渲染结果。大家可以关注下方指标。

每来一个 chunk，传统方案看到的仍然是当前的完整文档，所以它会重新解析全文、重新生成页面，再把结果写回 DOM。回答比较短时，这个成本不明显；但随着文本越来越长，前面已经稳定的内容也会被反复处理。

这里最重要的不是某一次渲染用了多少毫秒，而是计算量会随着文档长度和 chunk 数同时增长。

【如果 Demo 没有正常播放】这里的实现逻辑很简单：每个 chunk 到来后，都对累计全文执行一次 render；所以即使不看动画，大家也可以从这个更新路径看到它的重复工作。

【过渡】如果内容只有普通段落，问题还不算最严重。真正把差距放大的，是代码高亮和复杂内容块。

## 第 7 页｜从文本到丰富内容

为了让用户更直观地理解 AI 生成的内容，我们不能只把所有东西显示成纯文本。代码需要语法高亮，图片需要加载和布局，数学公式需要排版，Mermaid 还要从描述生成真正的图表。

这些内容被真正渲染出来以后，确实更容易阅读，但一次更新要做的事情也更多了。每来一个 chunk，不仅可能重新解析全文，还可能再次处理代码、公式和图表，然后更新页面，触发布局和绘制。

如果用户此时还在滚动、选择文字或输入，这些工作就会和页面交互争用主线程。

所以丰富内容本身不是问题，它们是 AI 产品体验中很重要的一部分。真正的问题是：在流式输出过程中，这些处理会不会随着每个 chunk 被全量重复执行。

【过渡】下一页我们加入 code、diff、Mermaid 和 KaTeX，再看一次全量渲染的过程。

## 第 8 页｜Demo：复杂内容放大全量成本

【操作】点击“播放全量”，让动画运行几秒；讲到各种复杂内容时，可以用鼠标指向对应区域。

现在这组测试内容里不只有普通文本，还包含 TypeScript、diff、Mermaid 图表和 KaTeX 公式。它们很接近真实 AI 产品中的输出。

可以看到，当这些内容块还没有闭合、内容还在持续变长时，全量渲染会不断重复高亮、图表处理、公式处理和 DOM 更新。越靠后的一个小 chunk，越可能让前面已经完成的复杂内容再次参与计算。

因此真正的问题不是“Markdown 能不能渲染这些内容”，而是“在 stream 过程中，能不能只处理真正发生变化的部分”。

【如果 Demo 没有正常播放】这一页想表达的是：代码、图表和公式的单次处理成本更高，而全量渲染会把这份成本乘上 chunk 数；后面的性能页会给出离线回放数据。

【过渡】把刚才的过程拆开来看，就能理解为什么回答越长，页面越容易变慢。

## 第 9 页｜为什么内容越长，重复工作越多

传统方案在流式场景里的循环是：收到一段新内容，重新解析全文，重新渲染全文，再替换整块 DOM。

这里有两个因素会同时增加开销。第一，回答越长，每次重新处理全文要做的事情越多；第二，更新次数越多，这套流程被重复执行的次数也越多。所以后面即使只增加几个字符，也可能再次处理前面已经完成的整篇内容。

直接后果是抖动和闪烁；更隐蔽的成本是交互状态被破坏，比如滚动锚点漂移、选择丢失、组件重新挂载。

所以只让某一次渲染变快还不够，更重要的是减少不必要的重复工作，改变每次收到新内容时的更新方式。

【过渡】这就是 markstream-vue 的核心设计出发点。

## 第 10 页｜核心机制

markstream-vue 的核心可以概括成三件事。

第一，parser 只处理 dirty tail，也就是文档尾部仍然在变化的区域；前面已经完成的内容，不再随着每个 chunk 重复解析。

第二，在 Vue 层复用稳定节点。已经完成的段落、代码块或组件，不需要反复卸载和重建，只更新尾部发生变化的节点。

第三，把一次较大的渲染拆分到多个画面帧中执行。每一帧只处理一部分内容，优先保障滚动、选择和输入不卡顿。

一句话概括：我们不再把每个 chunk 当成一篇新文档，而是把它当成对当前文档尾部的一次增量修改。

【过渡】下一页继续使用同一组测试内容，只把渲染方式换成 markstream-vue。

## 第 11 页｜Demo：markstream-vue 增量渲染

【操作】点击“播放全量”，先让观众观察页面稳定性，再指向下方指标。

现在输入内容和刚才相同，但更新方式变了：前面已经完成的内容直接复用，只重新处理仍在变化的尾部。

大家可以注意两个现象。第一，前面已经完成的内容会保持稳定，不需要随着尾部增长反复重建。第二，一次较大的更新会被拆到多个画面帧中执行，因此滚动和输入不会被长时间阻塞。

这也是 markstream-vue 和普通 Markdown 组件最本质的区别：它优化的不是完成之后的那一次展示，而是内容从第一个 chunk 到最终完成的整个生命周期。

【如果 Demo 没有正常播放】即使现场环境影响了动画，核心路径也不变：复用前面已经完成的内容，只重新解析并更新 dirty tail；后面的性能数据来自同一组测试内容的本地回放，不依赖这次现场演示。

【过渡】如果大家想在项目里尝试，基础接入其实非常轻。

## 第 12 页｜快速开始

最简单的用法只需要一个组件和两个核心 props。

先从 markstream-vue 引入 MarkdownRender；把模型当前累计输出绑定给 content；流结束后，再通过 final 告诉组件，内容已经输出完成。

左边就是完整的基础代码，右边是渲染效果。普通段落、链接、列表、引用、表格和代码块，都可以沿用熟悉的 Markdown 使用方式。

也就是说，使用方不需要先理解整套增量 parser，只需要把现有的 stream 状态接进来，就可以开始验证。

【过渡】刚才我们看到的是页面表现。接下来使用同一组包含代码、diff 和 Mermaid 的测试内容，对比优化前后的实际耗时。

## 第 13 页｜复杂内容的性能对比

这一页测试的是复杂内容持续输出时的性能。环境是 Chrome 150、Vue 3.5.34 和 Apple M1 Pro；每一项包含 119 次增量更新，每组运行三次并取中位数。

先看脚本执行耗时。diff 从 270 毫秒下降到 50 毫秒，减少约 81%；TypeScript 从 247 毫秒下降到 51 毫秒，减少约 79%；Mermaid 从 288 毫秒下降到 51 毫秒，减少约 82%。与此同时，DOM 数量和平均更新时间也明显下降。

比具体数字更重要的是这些变化从哪里来：前面已经完成的内容不再重复解析，稳定节点不再反复重建，页面布局也不会被每个小更新大范围影响。它改变的是更新方式，而不是简单地让同一次全量渲染跑快一点。

这组数据不是对所有 Markdown 内容的笼统承诺。当前重复表格的正确性检查还没有通过，所以没有放进结果；这里展示的只有已经通过检查的 TypeScript、diff 和 Mermaid。性能优化的前提始终是正确性。

【过渡】前面这些优化，解决的是怎样减少重复解析和重复渲染。但只减少重复工作还不够，真正需要解析的内容也要处理得足够快。为此，我们专门针对流式增量解析场景设计了 markdown-it-ts。从一开始，它就把大文本和增量输入作为性能目标，这也是它能够更快的核心原因。

## 第 14 页｜更快的底层解析

这一页主要看 markdown-it-ts 的解析性能。为了让不同实现可以在同一个口径下比较，这里先使用 Markdown 到 HTML 的完整渲染 API。我们把 markdown-it-ts 和另外五种常见实现放在同一环境下，输入相同的 20 万字符。测试环境是 Node 24.16 和 Apple M1 Pro，耗时越低越好。

在这组测试里，markdown-it-ts 的完整 render 大约是 0.70 毫秒，是当前最快的一档。对比原生实现 @ox-content/napi，在不同文本规模下大约快 1.66 到 2.69 倍；对比 markdown-it，大约是 10.92 到 16.14 倍。

这里的重点不是宣布一个永远不变的排行榜，不同库的目标和 API 范围并不完全相同。更重要的是，markdown-it-ts 并不是先做一个面向完整文档的通用 parser，再额外增加流式能力；流式增量解析的需求从一开始就进入了它的设计。即使在这一页的完整渲染测试中，这种设计依然带来了更短的处理路径和更低的耗时。

【过渡】但 parser 再快，也不能独占浏览器主线程，所以还需要调度层。

## 第 15 页｜将渲染拆分到多帧

在 60 帧的页面里，一帧只有 16.7 毫秒，而且这不是 Markdown 的专属预算。浏览器还要处理输入、滚动、布局、绘制，以及页面上的其他组件。

markstream-vue 在这一层提供了几个控制点：解析结果可以在 Worker 或 SSR 中处理后传回页面；长文可以限制页面中同时存在的活跃节点数量；较大的更新也可以分成多个小批次执行。

这里的原则很简单：用户交互优先，Markdown 使用剩余时间。因为对于用户来说，回复晚几十毫秒完成通常可以接受，但滚动和输入突然卡住，会马上被感知。

【过渡】解决更新性能以后，我们还可以再往前一步：AI 输出其实已经不只是 Markdown 文本。

## 第 16 页｜AI 输出是一种 UI 协议

真实的 Agent 或 AI 应用里，输出首先会包含段落、列表、代码块、表格这些常规 Markdown 结构，也可能同时包含 thinking、tool-call、tool-result 和业务组件。

如果我们只把这些内容当成字符串，就只能不断拼 HTML；但如果把它们看成协议里的不同 block，就可以映射成真正的 Vue 组件，各自拥有状态、交互和样式。

所以在 AI UI 里，Markdown 不只是文本格式，也可以作为一种统一的内容结构：文本、状态、工具结果和业务组件，可以在同一条 stream 里一起传输，再由前端组合成稳定界面。

【过渡】markstream-vue 为这种场景保留了自定义接口。

## 第 17 页｜高级 API

如果只是渲染普通 Markdown，前面的 content 和 final 已经够用。需要接 Agent 协议时，可以再打开这几个配置项。

通过 setCustomComponents，可以把 thinking、tool-result 这样的标签映射到 Vue 组件；custom-html-tags 用来告诉 parser 哪些标签需要按自定义 block 处理；batch-rendering 和相关参数负责控制主线程预算；max-live-nodes 则用来控制长文场景中的活跃节点规模。

这套 API 的目标不是重新发明一套业务协议，而是让你现有协议中的结构化内容，可以安全地进入 Vue 渲染层。

【过渡】放到整个技术栈里看，markstream-vue 的位置其实很明确。

## 第 18 页｜生态中的位置

markdown-it 和 Shiki 很适合完成态文档与代码高亮；stream parser 和 markdown-it-ts 负责处理大文本与流式增量输入；markstream-vue 再把解析结果持续更新成稳定的 Vue 界面；再往上，就是业务自己的工具、图表、引用等组件。

这里的 parser 和 Vue 渲染层是分开的，所以底层能力并不只属于 markstream-vue。parser 可以独立接收流式 Markdown，输出结构化结果，再交给不同的界面去展示。除了浏览器里的 Vue 应用，它也可以用于 TUI，也就是终端用户界面等非浏览器场景。

在 Web 端，我们也基于同一套 parser 扩展出了 markstream-react、markstream-angular 和 markstream-svelte。不同框架复用相同的流式解析能力，再接入各自的组件渲染层。

所以 markstream-vue 并不是要替代整个 Markdown 生态。它负责的是 AI stream 到 Vue UI 之间的增量渲染；同一套底层 parser 则可以服务不同前端框架，以及 TUI 等更广泛的界面形态。

【过渡】底层 parser 可以进入不同的界面形态，上层的 markstream-vue 也已经被不同领域的开源软件采用。

## 第 21 页｜结束与 Q&A

当 AI 输出变得更快、更长、更复杂，我们不能只是把静态 Markdown 更频繁地渲染，而要为 stream 设计新的更新模型。

markstream-vue 的答案是：只处理仍在变化的尾部，复用已经稳定的节点，并把较大的渲染拆分到多个画面帧中执行。

我的分享就到这里。项目地址是 GitHub 上的 Simon-He95 / markstream-vue，也可以扫描右侧二维码。谢谢大家，欢迎提问。

## 上台前 1 分钟速记

如果现场紧张，不需要背整篇，只记住下面五个节点：

1. **开场问题**：传统 Markdown 面向完成稿，AI 输出是持续变化的中间态。
2. **三个症状**：重复 parse、DOM 增长、交互抖动。
3. **成本模型**：chunk + 1 → parse all → render all → replace all。
4. **三个机制**：只处理变化中的尾部、复用稳定节点、将渲染拆分到多帧。
5. **结尾判断**：AI stream 需要新的更新模型，不是更多次静态渲染。

三个 Demo 页如果时间不够，每页只说一句：

- 第 6 页：传统方案对每个 chunk 都重新处理完整文档。
- 第 8 页：代码、图表和公式会把全量渲染成本继续放大。
- 第 11 页：markstream-vue 复用已经完成的内容，只更新仍在变化的尾部。
